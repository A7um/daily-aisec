# AI 安全每日观察：当治理框架遇上现实漏洞

*2026年2月27日*

---

## 今日要点

今天的 AI 安全领域有几个重要进展：

- UC Berkeley 发布了首个 AI Agent 风险管理框架[^1]
- Octane Security 用 AI 发现了 Nethermind 的高危漏洞[^2]
- 安全研究人员在 5 大 AI 编程工具中发现 69 个漏洞[^3]
- NIST 启动 AI Agent 标准倡议，征求公众意见[^6][^7]
- Vibe coding 从 meme 演变为有文档记录的现象，同时引发安全和认知债务讨论[^11][^14][^15]

## 一、UC Berkeley：首个 Agent 风险管理框架

### 为什么需要专门的框架？

2月24日，UC Berkeley 长期网络安全中心（CLTC）发布了《Agentic AI Risk Management Standards Profile》[^1]——首个专门针对自主 AI 系统的风险管理框架。这份框架由 CLTC 的 AI 安全倡议（AI Security Initiative）编写，扩展了 NIST 的 AI 风险管理框架（AI RMF），为自主系统提供了针对性指导[^4]。

**核心洞察：** "风险是自主系统的涌现属性，而不仅仅是单个模型的属性。"[^1]

这句话揭示了当前 AI 安全讨论的盲区。我们一直在讨论模型安全（如何防止 LLM 生成有害内容），但当模型被嵌入到一个**可以自主规划、执行任务、访问外部工具**的系统中时，风险的性质发生了根本变化。

**举个例子：**
- 模型风险：Claude 生成了一段恶意代码
- 系统风险：Claude Code 自动执行了那段代码，删除了你的生产数据库

前者是输出问题，后者是**权限 × 自主性**的问题。

### 四个治理支柱

框架提出了四个关键治理维度[^1]：

1. **可见性（Visibility into action）**：完整的活动日志和推理可追溯性，追踪整个决策序列和工具使用，而不仅仅是最终输出
2. **人机协作（Human-machine teaming）**：根据风险等级设置不同的监督强度——清晰的升级路径和检查点，高风险操作需要人类批准
3. **权限管理（Authority and permission scope）**：严格的权限控制、模拟环境和压力测试，部署前必须完成
4. **攻击面扩展（Expanded attack surface）**：Agent 特有的漏洞——它们难以区分系统指令和外部数据，这是与传统软件不同的脆弱性

**Agency 的光谱：** 框架认识到 agency 不是二元分类，而是一个光谱。治理要求应该与系统的自主性成比例[^1]。

**Agent 卡片：** 类似于模型卡片（model cards），这些结构化文档格式将捕获系统限制、来源和部署能力[^1]。

### 2月11日专家研讨会

2月11日的网络研讨会汇集了来自 Cooperative AI Foundation、Center for the Governance of AI 和 World Economic Forum 的专家[^1]。他们的共识是："随着 Agent 获得自主性、权限和对现实世界系统的访问，风险管理必须相应演进。"

**关键挑战：** Alan Chan（GovAI 研究员）在研讨会上指出，用户经常为了加速工作流而选择"dangerously skip permissions"（危险地跳过权限检查），无意中制造了巨大的安全缺口[^1]。安全配置错误在 2025 年 OWASP Top 10 中排名第二大威胁[^5]。

### NIST 也在行动

**NIST 已启动 AI Agent 标准倡议**，正在征求公众意见，定义什么是 Agent、它能做什么、以及当它出错时谁负责[^6][^7]。这是首次有国家级机构针对自主 AI 系统的安全风险寻求行业输入。

多个 Twitter 消息源确认了这一动态[^6][^7]，标志着 Agent 治理从学术讨论进入政策制定阶段。

## 二、AI 发现了人类找不到的漏洞

就在学术界讨论治理框架的同时，工业界证明了 AI 在安全领域的实际价值。

### Nethermind 高危漏洞

2月26日，**Octane Security**——一家自称"AI 原生安全公司"的机构——宣布其 AI 工具发现了 Nethermind 的高危漏洞[^2]。Nethermind 是运行近 40% 以太坊验证者的软件。该漏洞在被利用前就被修复了，但影响深远。

**发现过程：** 在 Gnosis 和 Lido 赞助的审计竞赛中（为以太坊 Fusaka 升级做准备），Octane 与匿名研究员 Guhu 合作。他们的 AI-人类团队提交了 17 个问题，其中 16 个被修复，9 个被认为是严重问题[^2]。

**漏洞细节：** 黑客可以通过提交"格式错误的交易"（malformed transaction）来破坏验证者，导致所有基于 Nethermind 的提议者持续错过区块（sustained missed slots）。这会降低网络活跃度和可用性，同时验证者会错过区块奖励并受到惩罚[^2]。

**奖励：** Octane 在竞赛中排名第四，获得 70,633 美元。以太坊基金会额外奖励了 50,000 美元的漏洞赏金[^2]。

### AI 加速漏洞研究

**Octane CEO Giovanni Vignone 的声明：**

> "AI 已经极大地加速了漏洞研究。漏洞假设、漏洞利用验证和生产级报告现在可以快 10 倍完成，这改写了每个将代码放到链上的组织的威胁模型。"[^2]

**关键警告：** "如果你不使用 AI 持续发现和修复漏洞，你就是在与正在使用 AI 的黑客竞争。"[^2]

### 讽刺的时机

这个消息发布五天前，Anthropic 刚刚发布了一个新的安全工具，"扫描代码库以查找安全漏洞并建议有针对性的软件补丁"，导致网络安全股票下跌[^8]。

而就在几周前，AI 生成的代码让 Moonwell 协议用户损失了 270 万美元——那段代码已经通过了加密安全公司 Halborn 的审计[^9]。

**悖论：** 我们用 AI 加速开发，但也在加速制造漏洞。同时，AI 也在加速发现漏洞。正如一位 Twitter 用户所说："AI 正在编写漏洞利用代码。Metasploit 模块、CVE 概念验证、模糊测试工具。帮你调试代码的同一个模型也在帮别人找你代码中的漏洞。"[^10]

## 三、Vibe Coding：从 Meme 到百科全书（再到安全审计）

### 维基百科词条

**Wikipedia 现在有了 vibe coding 词条。**[^11] 19 小时前发布，标志着 vibe coding 从 meme 转变为有文档记录的现象。

但真正的故事是安全研究人员在深入研究时发现了什么。

### 69 个漏洞

新的安全研究在**5 大 AI 编程工具中发现了 69 个漏洞**[^3]。这证实了许多人的怀疑：**速度是有代价的。**

社区的反应是转向结构化方法：

**Prompt contracts（提示契约）** 正在成为 vibe coding 的结构化替代方案[^12]。一位开发者记录了这一转变："我停止了 vibe coding，开始使用 prompt contracts。Claude Code 从赌博变成了交付。"[^12]

**Vibe modeling（氛围建模）** 是另一个获得关注的新概念[^13]——通过视觉领域建模来改进 AI 生成的代码。核心思想：给 AI 更好的脚手架，获得更好的输出。

### 认知债务的讨论

Twitter 上关于"认知债务"（cognitive debt）的讨论激增[^14][^15][^16]。Matt Pocock 的一条推文引发了广泛共鸣："如果你把 AI 扔给一个糟糕的代码库，你会得到更糟的结果。垃圾进，垃圾出。而在脑海中维持它会让你陷入认知债务。"[^14]

Kevin Yan 指出："这就是为什么迭代提示对于复杂构建来说是一个陷阱。每条新消息不仅仅是添加信息——它添加噪音。模型不是在精炼，而是在杂耍。每一轮，它都会掉更多的球。你不是在协作，你是在制造认知债务。"[^15]

Lagoon Labs 总结了转变："AI Agent 已经超越了 vibe coding，转向了具有纪律性约束、护栏和生产就绪实践的 agentic engineering，这减少了认知债务并加速了现实世界的影响。"[^16]

### 基础设施正在建设

这与昨天关于认知债务的讨论一致，但焦点已经从"这可持续吗？"转移到"我们如何让这变得安全？"

- **Qodo.ai** 发布了最佳 AI 编程助手工具的比较[^17]
- **SitePoint** 发布了 vibe coding 和 AI 开发的综合指南[^18]
- **Product Hunt** 现在有了专门的 vibe coding 类别[^19]
- **roadmap.sh** 推出了 vibe coding 学习路径[^20]

基础设施正在建设中。问题是治理能否跟上步伐。

## 四、新工具与平台

### MiniMax：Expert 2.0 和 MaxClaw

MiniMax 推出了 **Expert 2.0** 和 **MaxClaw**，简化了 AI Agent 的创建和云部署[^21]：

- 即时部署（无需服务器/Mac mini/VPS）
- 24/7 运行在 Telegram/WhatsApp/Slack/Discord
- 降低了 Agent 部署的技术门槛

### OpenKoi v2026.2.25

**OpenKoi** 是一个用 Rust 编写的自学习 AI Agent，零依赖，与 OpenClaw 兼容[^22]。

**特点：** "OpenKoi improving OpenKoi"——自我改进的 Agent。已有近 200 次下载[^22]。

### Intent by augmentcode

定位为"下一代开发者工作空间"，具有多 Agent 编排和"Living Spec"方法[^23]。

### Agentfy for iOS

为 AI Agent 状态提供锁屏通知，解决"17 个 Agent 太多难以管理"的问题[^24]。

### AWS Security Agent

AWS 发布了 Security Agent——一个用于自动化渗透测试的多 Agent 架构[^25]。这标志着主流云服务商开始将 Agent 技术应用于安全领域。

### Block 的 Goose

Block 刚刚裁员 4,000 人，同时发布了公司历史上最好的季度业绩，股价上涨 23%。背后的原因是他们构建了一个名为 Goose 的开源 AI Agent（基于 Anthropic 的 Model Context Protocol），并在整个公司部署[^26]。

## 五、治理与部署现状

### 框架与标准

UC Berkeley 的框架提供了全面的指导[^1]，NIST 正在征求公众意见制定标准[^6][^7]。但与此同时，开发者正在发布带有"dangerously skip permissions"标志的 Agent，用户为了加快速度正在点击它们[^1]。

### 企业需求

Reddit 上的讨论显示，企业在部署 Agent 前正在要求治理工具[^27]。一位用户问："你们目前如何处理 AI Agent 的治理和合规性？"这表明实际部署中的治理需求正在上升。

### 多 Agent 系统的复杂性

多 Agent 编排平台正在涌现（qBotica、AutomationEdge）[^28][^29]，但协调问题仍未解决。一位开发者指出："让另一个 Agent 在使用不同工作树后解决合并冲突，是在错误的层面上解决多 Agent 协调问题。我们都在修复症状，而不是首先避免冲突。"[^30]

### 当前挑战

Nethermind 的发现证明 AI 可以帮助发现漏洞[^2]，但 Moonwell 事件和 69 个编程工具漏洞[^3][^9]也显示了风险的真实性。Agent 技术正在快速部署，治理基础设施需要跟上步伐。

---

## 结语

今天的动态涵盖了 AI 安全领域的多个方面：治理框架的发布[^1]、实际漏洞的发现[^2]、新工具的推出[^21][^22][^23][^24]，以及开发实践的演进[^11][^12][^14][^15]。

Agent 技术还在早期阶段，AI 也在帮助我们发现问题[^2]。但随着更多 Agent 被部署、更多权限被授予，建立有效的治理基础设施变得越来越重要。
